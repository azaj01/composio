We've strengthened how credentials are protected on Composio-managed auth configs. For connections that use one of Composio's default toolkit integrations, access tokens, refresh tokens, and the `Authorization` header are now always masked in `connected-accounts` API responses — independent of your project's `mask_secret_keys_in_connected_account` setting.

**Most integrations need no changes.**

- Tool execution through the SDK, Tool Router, and MCP is unchanged. Composio continues to handle authentication server-side, so your agents and workflows never needed to read these tokens directly.
- Connections on your own auth config are unchanged. Token visibility continues to follow your project's masking setting exactly as before.

## What is a Composio-managed auth config?

When you connect a user through a toolkit's default integration — for example, Gmail or Slack without registering your own OAuth application — the connection is authorized through an OAuth application that Composio maintains on behalf of every team using that default. The access and refresh tokens the provider issues for that connection are bound to Composio's shared OAuth client.

You can tell a connection is Composio-managed by the `auth_config.is_composio_managed: true` field on the connected-account resource.

## Why we're making this change

Because the OAuth application is shared across every team on that default, any action a provider takes on it — rate limits, scope adjustments, or in the extreme case, suspending the OAuth client — is felt by everyone using it. A shared application means shared outcomes.

Keeping these credentials inside Composio removes an avenue of risk that no single customer can contain on their own, and lets us keep the default integrations healthy and available for every team that relies on them. Tool execution flows continue to work without any change; Composio uses the tokens server-side so your agents don't need to.

## What changes in the API

| Connection type | Project masking setting | Tokens in response |
|---|---|---|
| Composio-managed | On | Masked (unchanged) |
| Composio-managed | Off | **Masked (new behavior)** |
| Your own auth config | On | Masked (unchanged) |
| Your own auth config | Off | Returned in full (unchanged) |

### Example

Before, with project masking disabled on a Composio-managed Gmail connection:

```json
{
  "state": {
    "val": {
      "access_token": "ya29.a0ARrdaM9...full token...",
      "refresh_token": "1//04...full token..."
    }
  }
}
```

After:

```json
{
  "state": {
    "val": {
      "access_token": "ya29...",
      "refresh_token": "1//0..."
    }
  }
}
```

The same treatment applies to the `data`, `params`, and `headers.Authorization` fields wherever these values would otherwise appear.

### Affected endpoints

- [`GET /api/v3/connected_accounts/{id}`](https://docs.composio.dev/rest-api/connected-accounts/get-connected-accounts-by-nanoid)
- [`GET /api/v3/connected_accounts`](https://docs.composio.dev/rest-api/connected-accounts/get-connected-accounts)
- [`POST /api/v3/connected_accounts`](https://docs.composio.dev/rest-api/connected-accounts/post-connected-accounts)

## If your workflow reads tokens directly

A small number of workflows — typically those that call provider APIs outside of Composio's execution path — read `access_token` or `refresh_token` from the connected-accounts response. If that's you, the recommended path is to connect through an auth config you own. Tokens issued against your OAuth application belong to you, and the connected-accounts API returns them in full (subject to your project's masking setting).

Getting set up:

1. Register an OAuth application with the provider — for Gmail, you can start in the [Google Cloud Console](https://console.cloud.google.com/apis/credentials).
2. [Create a custom auth config](https://docs.composio.dev/rest-api/auth-configs/post-auth-configs) using those credentials.
3. Create new connections against that auth config. Response fields behave as they always have for customer-owned configs.

<Callout type="info">
**No action is needed for most teams.** If you build on the Composio SDK, Tool Router, or MCP, your integrations keep working as-is. If your application reads `access_token` or `refresh_token` directly from a Composio-managed connection, moving those connections onto a custom auth config preserves the previous response shape.
</Callout>
