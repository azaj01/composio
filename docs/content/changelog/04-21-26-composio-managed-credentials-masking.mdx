title: "Automatic Credential Masking for Composio-Managed Connections"
description: "Access and refresh tokens for connections on Composio-managed auth configs are now always masked in API responses, regardless of project-level masking. Connections on your own auth config are unchanged."
date: "2026-04-21"
---

For connections created through a Composio-managed auth config, access tokens, refresh tokens, and the `Authorization` header are now always masked in `connected-accounts` API responses — regardless of your project's `mask_secret_keys_in_connected_account` setting.

**Most integrations need no changes.**

- Tool execution through the SDK, Tool Router, and MCP is unchanged. Composio handles authentication server-side, so your agents never needed to read these tokens directly.
- Connections on your own auth config are unchanged. Token visibility continues to follow your project's masking setting.

### What is a Composio-managed auth config?

When you connect a user through a toolkit's default integration — for example, Gmail or Slack without registering your own OAuth application — the connection runs through an OAuth application that Composio owns and operates on behalf of every team using that default. The access and refresh tokens the provider issues are bound to Composio's shared OAuth client.

You can identify a Composio-managed connection by the `auth_config.is_composio_managed: true` field on the connected-account resource.

### Why this change

These default integrations run on OAuth applications that Composio maintains and that every team on the default shares. If a provider takes enforcement action on one — rate limits, scope restrictions, or suspending the client outright — the integration goes down for every team on it at the same time.

Raw tokens from `connected-accounts` responses carry that shared OAuth application's identity, which meant a single misuse anywhere could affect every other team relying on the same default. Keeping the tokens inside Composio — where your tools already execute them — removes that exposure without changing how anything works for you.

### Example

Before, with project masking disabled on a Composio-managed Gmail connection:

```json
{
  "state": {
    "val": {
      "access_token": "ya29.a0ARrdaM9...full token...",
      "refresh_token": "1//04...full token..."
    }
  }
}
```

After:

```json
{
  "state": {
    "val": {
      "access_token": "ya29...",
      "refresh_token": "1//0..."
    }
  }
}
```

The same applies to `data`, `params`, and `headers.Authorization` wherever these values would otherwise appear.

- [`GET /api/v3/connected_accounts/{id}`](/reference/api-reference/connected-accounts/getConnectedAccountsByNanoid)

- [`POST /api/v3/connected_accounts`](/reference/api-reference/connected-accounts/postConnectedAccounts)

### If you need raw tokens

To keep reading `access_token` or `refresh_token` from API responses, connect through your own auth config. Tokens issued against an OAuth application you own are returned in full (subject to your project's masking setting).

1. Register an OAuth application with the provider — for Gmail, start in the [Google Cloud Console](https://console.cloud.google.com/apis/credentials).
2. [Create a custom auth config](/reference/api-reference/auth-configs/postAuthConfigs) using those credentials.
3. Create new connections against that auth config.
