title: "Automatic Credential Masking for Composio-Managed Auth Configs and Connections"
description: "Auth-config and connected-account API responses now redact credentials for Composio-managed resources and mask them for customer-owned ones — a fail-closed allowlist replaces the previous blacklist for both."
date: "2026-04-21"
---

Two API surfaces now fail closed when returning credential-bearing fields, replacing two previously independent blacklists with a single allowlist policy:

- **`auth_configs` GET responses** — `client_secret`, `consumer_secret`, `bot_token`, `bearer_token`, `verification_token`, `developer_token`, and any future sensitive field are hidden. Composio-managed configs return `REDACTED`; configs you own return `<first4>...<last4>`.
- **`connected_accounts` GET / list / create responses** — `access_token`, `refresh_token`, `id_token`, and the `Authorization` header are hidden for connections on Composio-managed auth configs, regardless of your project's `mask_secret_keys_in_connected_account` setting.

**Most integrations need no changes.**

- Tool execution through the SDK, Tool Router, and MCP is unchanged. Composio handles authentication server-side, so your agents never needed to read these tokens directly.
- Connections and auth configs that you own still return your credentials — masked when your project has masking enabled, raw when it doesn't.

### What is a Composio-managed auth config?

When you connect a user through a toolkit's default integration — for example, Gmail or Slack without registering your own OAuth application — the connection runs through an OAuth application that Composio owns and operates on behalf of every team using that default. The `client_secret` of that application, and the access and refresh tokens the provider issues for connections on it, are bound to Composio's shared OAuth client.

You can identify a Composio-managed auth config or connection by the `is_composio_managed: true` field on the resource.

### Why this change

A single OAuth application backs every team on a default integration, and the credentials we return — both the `client_secret` on the auth config and the tokens on connections made through it — are bound to that one application. If a provider throttles, restricts, or suspends the app in response to misuse of one of its credentials, the integration becomes unavailable for every team on the default at once.

Under the previous blacklist design, every new field added to a default auth scheme shipped raw until someone remembered to add it to the mask list. The most recent example was the Google Ads `developer_token`. The allowlist flips this: new fields are masked by default, and classifying a field as safe requires an explicit code change.

### Auth configs — what changed in responses

For `GET /api/v3/auth_configs` and `GET /api/v3/auth_configs/{id}`, the `credentials` object now behaves like this:

| Field | Custom auth config (`is_composio_managed: false`) | Composio-managed (`is_composio_managed: true`) |
|---|---|---|
| `scopes`, `user_scopes`, `scope`, `user_scope` | Pass through | Pass through |
| `oauth_redirect_uri`, `token_url`, `base_url` | Pass through | Pass through |
| `permissions`, `user_agent`, `client_name`, `logo_uri` | Pass through | Pass through |
| `full`, `region`, `subdomain`, `shop`, `hostUrl`, `version`, `prefix`, `suffix` | Pass through | Pass through |
| `client_id`, `consumer_key` | Pass through (you own it) | `REDACTED` (Composio owns it) |
| `client_secret`, `consumer_secret`, `code_verifier` | `<first4>...<last4>` | `REDACTED` |
| `bot_token`, `bearer_token`, `verification_token`, `generic_id`, `developer_token` | `<first4>...<last4>` | `REDACTED` |
| Any future unlisted field | `<first4>...<last4>` | `REDACTED` |

Strings shorter than 8 characters fall back to `REDACTED` instead of partial disclosure.

### Connected accounts — what changed in responses

For `GET /api/v3/connected_accounts`, `GET /api/v3/connected_accounts/{id}`, and `POST /api/v3/connected_accounts`, the `state.val`, `data`, `params`, and `headers.Authorization` fields are now always masked for connections on Composio-managed auth configs — regardless of your project's `mask_secret_keys_in_connected_account` setting.

Before, with project masking disabled on a Composio-managed Gmail connection:

```json
{
  "state": {
    "val": {
      "access_token": "ya29.a0ARrdaM9...full token...",
      "refresh_token": "1//04...full token..."
    }
  }
}
```

After:

```json
{
  "state": {
    "val": {
      "access_token": "ya29...",
      "refresh_token": "1//0..."
    }
  }
}
```

Connections on auth configs you own continue to follow your project's masking setting as before — raw tokens when masking is off, truncated when it's on.

### If you need raw credentials

If your workflow reads `client_secret` from an auth-config response, or `access_token` / `refresh_token` from a connected-account response, own the auth config yourself. Secrets on your own OAuth application are returned in full (subject to your project's masking setting).

1. Register an OAuth application with the provider — for Gmail, start in the [Google Cloud Console](https://console.cloud.google.com/apis/credentials).
2. [Create a custom auth config](/reference/api-reference/auth-configs/postAuthConfigs) using those credentials.
3. Create new connections against that auth config.
