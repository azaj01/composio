Composio supports two ways to authenticate users with toolkits. You can use them together, and most production apps do.

- **Composio managed apps**: Composio registers and maintains OAuth apps for popular toolkits (GitHub, Gmail, Slack, etc.). Zero setup, works out of the box.
- **Your own OAuth apps**: You register an OAuth app in the toolkit's developer portal and tell Composio to use your credentials instead.

## When to use Composio managed apps

Managed apps are the default. Every toolkit that supports OAuth has a Composio managed app ready to go. Use them when:

- **You're building and iterating.** No OAuth app registration, no credentials to manage. Create a session and start testing immediately.
- **Default scopes cover your needs.** Composio requests sensible defaults for each toolkit.
- **Branding on consent screens doesn't matter.** Users will see "Composio wants to access your account" during OAuth. Fine for internal tools, prototypes, and development.

<Tabs groupId="language" items={['Python', 'TypeScript']} persist>
<Tab value="Python">
```python
from composio import Composio

composio = Composio()
session = composio.create(user_id="user_123")
# GitHub, Gmail, Slack, etc. all use Composio managed auth by default
```
</Tab>
<Tab value="TypeScript">
```typescript
import { Composio } from '@composio/core';

const composio = new Composio();
const session = await composio.create("user_123");
// GitHub, Gmail, Slack, etc. all use Composio managed auth by default
```
</Tab>
</Tabs>

## When to use your own OAuth app

Bring your own credentials when any of these apply:

- **Your users see OAuth consent screens.** In production, users should see your app name, not "Composio." This is the most common reason to switch.
- **You need custom scopes.** Composio's default scopes may not include everything you need (e.g., write access to a specific Google API).
- **You're hitting rate limits.** Managed apps share quota across all Composio users. Your own app gets a dedicated quota.
- **You're connecting to a custom instance.** Self-hosted or regional variants (e.g., a private Salesforce subdomain) need their own OAuth app.
- **Enterprise customers require your branding end-to-end.**

<Steps>
<Step>
<StepTitle>Register an OAuth app with the toolkit</StepTitle>

Go to the toolkit's developer portal (e.g., [GitHub](https://github.com/settings/developers), [Google Cloud Console](https://console.cloud.google.com/)) and create a new OAuth app. Set the callback URL to:

```
https://backend.composio.dev/api/v3/toolkits/auth/callback
```

Copy your **Client ID** and **Client Secret**.
</Step>

<Step>
<StepTitle>Create a custom auth config in Composio</StepTitle>

In the [Composio dashboard](https://platform.composio.dev):

1. Go to **Authentication management** and click **Create Auth Config**
2. Select the toolkit (e.g., GitHub)
3. Choose the **OAuth2** scheme
4. Toggle on **Use your own developer credentials**
5. Enter your **Client ID** and **Client Secret**
6. Click **Create**

Copy the auth config ID (e.g., `ac_1234abcd`).
</Step>

<Step>
<StepTitle>Pass the auth config ID in your session</StepTitle>

<Tabs groupId="language" items={['Python', 'TypeScript']} persist>
<Tab value="Python">
```python
from composio import Composio

composio = Composio()
session = composio.create(
    user_id="user_123",
    auth_configs={
        "github": "ac_your_github_config",
        # toolkits not listed here still use Composio managed auth
    },
)
```
</Tab>
<Tab value="TypeScript">
```typescript
import { Composio } from '@composio/core';

const composio = new Composio();
const session = await composio.create("user_123", {
  authConfigs: {
    github: "ac_your_github_config",
    // toolkits not listed here still use Composio managed auth
  },
});
```
</Tab>
</Tabs>
</Step>
</Steps>

## Using both together

Most production apps mix both. Use your own credentials for toolkits where users see consent screens (GitHub, Google, Slack) and let Composio managed auth handle the rest.

<Tabs groupId="language" items={['Python', 'TypeScript']} persist>
<Tab value="Python">
```python
session = composio.create(
    user_id="user_123",
    auth_configs={
        "github": "ac_your_github_config",
        "google": "ac_your_google_config",
        # everything else uses Composio managed auth automatically
    },
)
```
</Tab>

});

### Moving from managed to custom

Start with managed apps during development. When you're ready for production, register your own OAuth apps for the toolkits that matter and add the `auth_configs` parameter. No other code changes needed.

Existing connected accounts keep working because they stay tied to the auth config they were created with. New connections go through your own credentials. See [Switching from managed to custom](/docs/white-labeling-authentication#switching-from-composio-managed-to-your-own-oauth-app) for details.

### Toolkits without managed auth

Some toolkits (PostHog, Tavily, Perplexity, etc.) don't have Composio managed auth. For these, you must create a custom auth config with your own credentials. See [Using custom auth configuration](/docs/using-custom-auth-configuration).

## FAQ

<Accordions>
<Accordion title="Can I use managed auth for some toolkits and custom apps for others?">
Yes. Pass `auth_configs` only for the toolkits where you want your own credentials. Everything else automatically uses Composio managed auth. This is the most common pattern in production.
</Accordion>

<Accordion title="Will switching to a custom app break existing connected accounts?">
No. Existing connected accounts are tied to the auth config they were created with and keep working. Only new connections go through your custom OAuth app. To fully migrate existing users, delete their old connected accounts and have them re-authenticate. See [Switching from managed to custom](/docs/white-labeling-authentication#switching-from-composio-managed-to-your-own-oauth-app).
</Accordion>

<Accordion title="Do I need a custom app for API key-based toolkits?">
No. API key-based toolkits (like Tavily, Perplexity) don't have OAuth consent screens, so there's no branding to customize. You still need to create a custom auth config to provide your API key, but you don't need to register an OAuth app.
</Accordion>

<Accordion title="Does Composio still handle token refresh with custom apps?">
Yes. Composio manages token refresh regardless of whether you use managed or custom OAuth credentials. You don't need to handle re-authentication or token expiration.
</Accordion>

<Accordion title="What happens if Composio's managed OAuth app hits rate limits?">
Managed apps share quota across all Composio users. If you're seeing rate limit errors, switching to your own OAuth app gives you a dedicated quota. This is one of the main reasons to use your own app in production.
</Accordion>

<Accordion title="Can I create custom auth configs programmatically?">
Yes. Use the SDK instead of the dashboard. See [Programmatic auth configs](/docs/auth-configuration/programmatic-auth-configs) for examples.
</Accordion>
</Accordions>
